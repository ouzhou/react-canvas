# react-canvas 与 OpenPencil 路线对比

本文从**底层技术选型**出发，对照 **react-canvas**（本仓库的 React + reconciler + Yoga + Skia）与 **OpenPencil** 一类 **文档驱动 + 引擎 + Skia** 的方案，便于产品定位与选型讨论。

> 说明：OpenPencil 的信息来自其公开文档/社区资料；具体实现以对应仓库为准。

---

## 选型差异（一句话）

|              | **OpenPencil 路线**                                       | **react-canvas 路线**                                        |
| ------------ | --------------------------------------------------------- | ------------------------------------------------------------ |
| **真相来源** | 可序列化 **`PenDocument` + 状态机/Store**，画布是「视图」 | **React 组件树 + reconciler → 场景树**，画布是「声明式 UI」  |
| **心智模型** | 设计工具：文档、页面、实例、变量、导入导出                | 应用 UI：组件、props、状态，与 React Native / 前端习惯接近   |
| **强项场景** | AI 改稿、`.op` 文件、Figma 互操作、编辑器工作流           | 在 canvas 上做 **产品界面**、强交互、与现有 React 栈拼在一起 |

---

## 优缺点摘要

### OpenPencil 类（文档 + 引擎 + Skia）

**优点**

- **持久化与协作叙事清晰**：一份 JSON 树，Git、diff、版本都好讲。
- **流式 / AI**：增量写文档（如向文档树插入节点）有天然落点。
- **画布超大、更新很密**时，可少让 React 参与整棵画布子树（若实现得当）。
- **设计工具能力**：主题变量、组件实例、导出代码等围绕「稿」建。

**缺点**

- **应用级 UI 开发体验**不如「全是 React」：复杂交互往往要绕 store/命令。
- **和 npm 组件生态**不是同一套复用方式（稿里的实例/引用，而非 `import Button`）。
- 自研布局/同步链路多，**工程复杂度高**。

### react-canvas（React + reconciler + Yoga + Skia）

**优点**

- **动态界面、条件渲染、局部状态**：用 React 即可；**事件 / 主题 / 组件复用**走熟悉路径。
- **和现有 React 工程、设计系统（Context、hooks）**容易拼。
- **开发者体验**：`View` / `Text` + `style`，偏 **React Native 式**，对前端友好。
- 产品若偏 **运行时 App UI** 而不是「存一份设计文件」，模型更直接。

**缺点**

- **没有**开箱即用的「行业设计文件格式」叙事（需自行定序列化或只做运行时）。
- **超大画布 + 极频繁更新**时，reconciler 与宿主节点成本要专门优化（分页、降频、批帧）。
- **AI 流式改整棵树**：要么接外部文档层，要么自行设计 patch，**不是文档优先模型**。

## 选型参考：什么情况下更适合 react-canvas

从**用户 / 产品**角度，若符合下面多条，**react-canvas 这类方案往往更合适**：

- 要做的是 **带复杂交互的界面**（表单、列表、手势、多状态机），并希望 **与 React 状态、路由、数据请求** 绑在一起。
- **主题切换、国际化策略** 希望跟 **组件树** 走，而不是跟独立设计稿文件格式强绑定。
- 希望 **复用团队已有的 React 组件与模式**（甚至逐步把一块 DOM UI 迁到 canvas），而不是以 **设计稿文件** 为中心协作。
- 产品 **不是**「AI 生成可编辑矢量设计稿 + 导出 + Figma 工作流」为主，而是 **canvas 上的应用 UI**、**可视化控制台**、**游戏式 HUD** 等。
- 接受：**序列化、多人协作稿、设计—开发对齐** 由上层产品自行定义，而不是默认对标 OpenPencil 全套能力。

## 与 OpenPencil「产品不一样」时如何表述

- **OpenPencil** 更接近：**设计工具 + Design-as-Code + AI 改稿**。
- **react-canvas** 更接近：**用 RN 式 API 在浏览器里画应用界面**。

二者**不是简单的竞品关系**，而是**不同问题域**：选 React 驱动，是为了 **应用 UI 与工程体验**；选文档驱动，是为了 **稿、导出与 AI 流式**。对外说明时，用「问题域不同」比「谁更强」更贴切。
