# 已知问题与平台边界

本文档说明 **React Canvas**（Skia / CanvasKit 绘制 + Yoga 布局 + React Reconciler）与浏览器 DOM/CSS 的差异，并区分：

- **缺陷（硬边界）**：在既定架构与目标下**无法**做到、或**明确不承诺**与 Web **等价**的能力；不是「还没写代码」的临时状态，而是选型与非目标本身。
- **非缺陷**：**可以通过实践解决**——包括路线图内的实现、工程优化、或 **DOM 混合**等成熟方案；归类为**待实现、可选增强或集成方式**，不记入「平台缺陷」。

实现进度以 [development-roadmap.md](./development-roadmap.md) 为准；交互语义以 [阶段三交互规格](./superpowers/specs/2026-04-05-phase-3-interaction-design.md) 为准。

---

## 1. 缺陷（硬边界）

下列条目在文档语境下视为 **缺陷或永久差异**，对外沟通时**不应**承诺「将来会与浏览器一致」。

| 缺陷                                       | 说明                                                                                                                                                                                                                          |
| ------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **完整 CSS 布局与 Web 对齐**               | 布局引擎为 **Yoga（Flexbox 语义，对齐 RN）**，**不是**浏览器布局。因此 **CSS Grid、`float` 等** 不在引擎能力内；除非更换或叠加另一套布局系统，否则**无法实现**与 Web 一致的完整布局模型。                                     |
| **运行时 CSS 选择器、层叠与伪类**          | **不实现**浏览器式的选择器、层叠、` :hover` / `:focus` 等**自动**求值。这不是遗漏，而是**非目标**；交互外观需用 **事件 + 状态 + `style`** 表达（见下文「非缺陷」）。                                                          |
| **与 Web 等价的 `className` / 原子类体验** | 若未来做构建期工具类，也只能映射到 Yoga/Skia **子集**，**无法**达到「写 Tailwind = 写 Web」的等价体验（见路线图阶段六 Step 16）。                                                                                             |
| **纯 Skia 场景树内的原生可编辑控件**       | 画布**绘制**文字，**不提供**与系统 `<input>` / `<textarea>` **同源**的宿主节点 API。在**不引入 DOM** 的前提下，**无法**获得与浏览器表单一致的原生可编辑与 IME 管线；**不是**靠多写几个 React 组件就能在纯 Canvas 里「补上」。 |
| **纯画布 API 的文件选择**                  | 无与 `<input type="file">` **等价**的 Skia/Canvas 调用；文件对话框依赖 **浏览器 / DOM**。                                                                                                                                     |

**小结：**「缺陷」指向 **模型能力**（布局/CSS/原生控件类别）与 **产品非目标**（运行时 CSS），而不是「我们还没写 ScrollView」这类工程 backlog。

## 2. 非缺陷（可通过实践解决）

下列条目**不**算作平台缺陷；解决方式包括 **core/react 路线图**、**应用或 UI 包中的实现**、或 **DOM 混合**。

| 事项                                              | 性质                | 可行实践                                                                                                                                      |
| ------------------------------------------------- | ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| **点击、悬停、按压、拖拽等交互**                  | 路线图 / 已实现方向 | DOM `pointer*` → 命中检测 → `onClick` / `onPointer*`；hover 用 `pointerenter`/`leave` 或 move + 命中链 + **状态改 `style`**（见阶段三规格）。 |
| **滚动、裁剪、惯性等**                            | 待实现能力          | `overflow`、`ScrollView`、`clip` 等与绘制/布局管线扩展（路线图阶段四等）。                                                                    |
| **图片展示、异步解码、重绘**                      | 待实现或演进        | `Image` 宿主、缓存与错误处理（见路线图）。                                                                                                    |
| **样式复用与合并**                                | 待实现能力          | `StyleSheet.create`、style 数组、`flatten`（路线图阶段六）。                                                                                  |
| **构建期 `className` → `style` 子集**             | 可选工程            | Babel/Vite 插件等；**不**改变「与 Web 不等价」的硬边界，但可改善书写体验。                                                                    |
| **文本输入、多行编辑**                            | **集成方案**        | **DOM 覆盖层**：在布局框上放置透明/定位的 `<input>` / `<textarea>`，同步几何与 `value`；属 **实践可解**，非 Skia 缺陷。                       |
| **文件选择**                                      | **集成方案**        | 隐藏 `<input type="file">` + 按钮触发 + 与状态同步；**实践可解**。                                                                            |
| **键盘 Tab 顺序、读屏、ARIA**                     | 路线图能力          | Proxy DOM / 无障碍层（路线图阶段六）；**实践可解**，非「画布像素能单独完成」。                                                                |
| **动画**                                          | 路线图能力          | 直接驱动场景树属性 + 脏标记 + rAF（路线图）；**实践可解**。                                                                                   |
| **命中精度（如圆角内忽略透明区、glyph 级 Text）** | 增强项              | 几何与遍历规则可迭代；当前简化实现属 **工程取舍**，非不可逾越。                                                                               |
| **字体加载、回退、排版规则**                      | 工程收敛            | Skia Paragraph + 项目规则与 [文字文档](./core/text-line-wrapping.md)；**可逐步对齐**，与「无 CSS 布局引擎」无关。                             |
| **性能与内存（WASM、大图）**                      | 工程优化            | 缓存、解码策略、降级路径；**可优化**，非单一「缺陷」条目。                                                                                    |

## 3. Yoga / RN 语义 vs Web：差异 ≠ 缺陷

以下情况属于 **与 React Native / Yoga 一致的行为或文档化差异**，**不是**实现错误：

- 默认 `flexDirection`、`flexShrink` 等与浏览器不同；
- 百分比、边距等相对 **Yoga 父节点** 的语义；
- 在已映射的 `ViewStyle` 子集内，布局结果应与 **RN 式约定**一致，而非强行对齐 MDN 上每一条 Web 特例。

若组件库以 **RN 式 `style`** 为主叙事，这些应写成 **「与 Web CSS 的差异」**，而不是 **「本项目的 bug 列表」**。

---

## 4. 对自建组件库（如 `@react-canvas/ui`）的提示

- **缺陷清单**用于设定期望：不要承诺 **完整 CSS**、**运行时伪类**、**纯画布原生表单控件**。
- **能力清单**应区分：**core 路线图内**（滚动、动画、无障碍…）与 **DOM 桥接组件**（输入、文件），后者是 **标准解法**，不是二等公民。
