# 事件系统：前因后果、参考模型与 Yoga 下的设计

**日期：** 2026-04-10  
**状态：** 说明文档（与 `core-design.md` §8 互补；实现细节以代码为准）

---

## 1. 为什么要单独想「事件系统」

在 **浏览器 DOM** 里，点击、悬停、滚动等由 **浏览器** 完成：命中谁、是否冒泡、是否穿透，都建立在 **盒模型 + 视图树 + 指针分发** 之上。

在 **Canvas 2D / Skia** 上绘制 UI 时：

- 没有原生「节点」与「命中」；
- 必须自己实现：**坐标变换 → 命中测试（hit test）→ 选定 target → 可选冒泡/捕获**；
- **叠放顺序**（谁盖住谁）若与 **命中顺序** 不一致，会出现「看得见 A、点到的却是 B」；
- **布局**若与 **叠放** 混在同一套「子节点顺序」里随意改动，又会和 **Flex/Yoga** 的语义冲突。

因此事件系统要解决的核心问题是：

1. **确定性**：同一帧内，**绘制顺序**与 **命中顺序**必须一致；
2. **层级**：多窗口/弹窗/下拉时，**上层必须能挡住下层**（同一次指针事件不应同时命中上下两层）；
3. **传播**：在 target 确定之后，**沿哪条路径**派发 `pointerdown` / `click` / 冒泡，且与 Yoga **布局树** 的关系要清晰；
4. **与 Yoga 分工**：Yoga 只负责 **几何与 flex 约束**；**谁在上**、**谁先被点** 由 **绘制/命中管线** 决定，而不是反过来要求 Yoga 模拟 DOM 的叠放上下文。

下文先说明 **Konva** 与 **React Native** 的常见做法，再归纳 **使用 Yoga 时更合适的设计**。

---

## 2. Konva 的做法（单画布场景树）

### 2.1 树与顺序

- **一棵场景树**：`Stage → Layer → Group → Shape`。
- **同一父节点下的 `children` 顺序**同时决定：
  - **绘制顺序**（越靠后越在上层）；
  - **命中顺序**（Layer 自顶向下；层内 hit canvas 像素对应「最上」的形状）。
- **`zIndex`**：在 Konva 里本质是 **调整 `parent.children` 的下标**，因此 **改 zIndex = 改树在兄弟间的顺序**。

### 2.2 命中与「不穿透」

- **`getIntersection`**：从 **上面的 Layer** 往下测；同一像素 **只返回一个** 最前的 `Shape`。
- **重叠区域**：下层兄弟 **不会** 作为同一次点击的 target；**冒泡**沿 **真实 `parent` 链** 向上，**不会**在重叠兄弟之间「横跳」。
- **弹窗**：通常 **更高 Layer** 或 **同层中 zIndex 更大的 Group**；若需点空白也不点到背后，常加 **全屏 Rect 遮罩**（`listening: true`）。

### 2.3 特点与边界

- **优点**：模型简单，**单一数据源**（`children` 顺序 = 绘制 = 命中）。
- **代价**：改叠放就要 **动兄弟顺序**；与 **Flex 布局**（兄弟顺序常表示 flex 流向）混在一起时，容易 **布局语义与视觉叠放耦合**。
- **与 CSS**：Konva **没有** 浏览器那种 **叠放上下文**；只在 **当前父节点的一群兄弟** 里比顺序，**不是**「去找最近的 `position: relative` 祖先再比 zIndex」。

---

## 3. React Native 的做法（原生视图树）

### 3.1 树与顺序

- **两棵相关树**：JS 侧的 React 树 + **原生侧** `UIView` / `Android View` 树；布局常用 **Yoga**（与 Web 的 CSS 盒模型部分对齐，但仍是 **RN 语义**）。
- **`zIndex`**：在 RN 中多为 **同父节点内** 的叠放顺序，**不一定** 通过「物理移动原生子视图数组」暴露给 JS，但 **触摸分发** 仍遵循 **视图层级上前者优先**。

### 3.2 弹窗与下拉

- **`Modal`**：常对应 **单独的窗口或最顶层容器**，在原生层级上 **盖在主界面之上**；触摸由 **系统** 先分发给最前层，背后界面 **默认收不到** 同一次触摸（除非特殊 `pointerEvents` 等）。
- **下拉 / 浮层**：常用 **`position: 'absolute'`** + **`measureInWindow`** 等算锚点，或用 **Portal** 挂到根；**点外部关闭** 依赖 **全屏 `Pressable` / 透明层** 或 **根级手势**。

### 3.3 特点

- **优点**：触摸由 **OS 视图系统** 处理，开发者较少自己写命中遍历。
- **与 Canvas 项目**：若在 **单张 Skia 画布** 上复刻 RN API，**没有** 原生视图树，必须在 core 里 **自研 hit test + 传播**，只能 **借鉴语义**（多 Layer、Modal、zIndex），不能照搬实现。

---

## 4. 我们要解决的具体问题（小结）

| 问题               | 含义                                                                           |
| ------------------ | ------------------------------------------------------------------------------ |
| **命中与绘制一致** | 用户看到的「最上」与「先被点到的」必须是同一套规则。                           |
| **弹窗不穿透**     | 打开弹窗时，背后按钮不应收到同一次 `pointerdown`/`click`。                     |
| **下拉锚点与滚动** | 浮层相对触发器定位；页面或列表滚动时要么 **重算位置** 要么 **关闭**。          |
| **传播路径清晰**   | 冒泡沿 **场景父链**；与「兄弟 z 序」解耦，避免误解成「改 z 就要换冒泡树」。    |
| **Yoga 友好**      | 尽量避免 **仅为叠放** 频繁重排 Yoga 子节点顺序，以免干扰 flex 语义与布局脏区。 |

---

## 5. 使用 Yoga 时更合适的设计（推荐）

以下与 **「结构树 vs 绘制/命中序」**、**「多根布局」** 相关，是本栈（Yoga + 单 Surface 绘制）下的推荐组合。

### 5.1 布局树与叠放解耦（首选）

- **结构 / Yoga**：`children` 顺序服务于 **flex 与数据/Reconciler 稳定性**，**不** 为叠放频繁 `splice`。
- **绘制与命中**：对子节点使用 **与 zIndex 一致的排序**（例如对「子节点 + 稳定次序」排序后遍历）；**绘制**与 **hit test** **共用同一排序函数**，保证一致。
- **效果**：与 Konva「改数组即改叠放」不同，但 **与 RN/CSS 里「文档顺序 + z-index」** 更接近；**改 zIndex 不必改 Yoga 树结构**。

### 5.2 多 Layer / 多 Yoga 根

- **同一 Yoga 实例**、**多个根节点**（多个 `Node.create()` 的根）分别 `calculateLayout`，是常见且推荐的做法；**不必**为每层单独起一个 Yoga WASM 实例。
- **Stage 多 Layer**：主界面、overlay、modal 各挂 **独立根**，**命中自顶向下**（zIndex 高的层先测），与 **绘制顺序** 一致。
- **「整层捕获」**（可选语义）：某层 `captureEvents === true` 时，本层 **即使未命中任何子节点** 也 **不再** 向低层继续命中，用于 **弹窗层点空白不穿透**。

### 5.3 弹窗（Modal）

- **结构**：弹窗内容放在 **顶层 Layer 或独立根**；**全屏遮罩**（可半透明）作为 **大块可命中区域**，避免「缝」里点到背后。
- **Yoga**：该层单独 **根布局**；无需与主界面 **共享同一 flex 根**。
- **预建节点**：非必须；首帧卡顿再考虑 **idle 预热**。

### 5.4 下拉（Dropdown）

- 与弹窗共享：**独立根 / 顶层叠放**。
- **额外**：**锚点**（触发器 `layout` → 换算到 Stage 坐标；贴边 **flip**）；**`maxHeight` + 内部滚动**；**外层滚动**时 **更新位置或关闭**。

### 5.5 事件传播（冒泡）

- **Target**：由 hit test **唯一**确定（最上可命中节点）。
- **冒泡**：沿 **场景节点 `parent` 链** 向上；**不** 在兄弟之间按 z 序传播。
- **与叠放**：重叠兄弟中只有 **被命中的** 进入传播链；**下层兄弟** 不会收到本次点击。

### 5.6 若未来要对齐 Web 叠放上下文

当前 **「父级内 zIndex + 多 Layer」** 已覆盖多数 RN 式 UI。若要对齐 **CSS 叠放上下文**（`position` / `opacity` / `transform` 等建树），需要 **单独规格**：哪些节点创建上下文、比较范围如何切分——**超出** 本文与默认 Yoga 映射范围时再立项。
